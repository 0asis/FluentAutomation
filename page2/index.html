Introducing the RemoteCommand API

I’ve mentioned the RemoteCommand API several times recently in posts, even on the Coming Soon portion of the site.

I think its time I explain what it is, why it exists and why I’m excited about it.

What

The RemoteCommand bits essentially expose a service endpoint that receives instructions in the form of a JSON document. The document contains a collection of Commands with the necessary Arguments to execute them.

Why

Any-Platform Test Creation

This allows for the origin of the tests to essentially be.. anything. Any language that can create well-formed JSON can run an automated UI test. We’re looking for contributors to help provide native fluent bindings for languages such as Ruby, PHP and Java.

An example of a Ruby test being built/executed via JSON is available here.

Playback from Fluent Automation Recorder

Another project that utilizes this API is the Fluent Automation Recorder by Paul Zumbrun. It allows you to instantly play back the test you’ve just recorded for verification.

Scalability

A common problem in UI Automation testing is the time involved in test runs. Anything that simulates user input is inherently slow compared to Unit Tests or even standard Integration Tests. As much as we try to speed things up, sometimes the only way to get through your tests in a reasonable manner is to scale out.

The native C# binding supports enabling Remote Execution which essentially targets a machine hosting the RemoteCommand service endpoint. This can be used for any number of scale out plans to spread the load of testing across several machines.

When

RemoteCommand support is in the master branch and in latest builds now, but not officially released. I’m working on the user experience of the FluentAutomation.Server (RemoteCommand endpoint) application to be useful in local debugging scenarios when tied with the Recorder.

I expect to have it out in the wild, ready to go in the next few weeks. The code is there for anyone who wants to try it out.

Let me know what you think of this upcoming feature!

Same Test, Multiple Browsers

When we added Selenium support a few weeks back, we finally had real support for multiple browsers. Quickly we decided that writing tests multiple times or even having to stub them out with different I.Use() calls just wasn’t a compelling enough experience.

Now… Enter the I.Record() and I.Execute() commands.

Take this simple test for example:

[TestMethod]
public void Drag_Drop_Test()
{
    I.Use(BrowserType.InternetExplorer);
    I.Open("http://developer.yahoo.com/yui/examples/dragdrop/dd-groups.html");
    I.Drag("#pt1").To("#t2");
    I.Drag("#pt2").To("#t1");
    I.Drag("#pb1").To("#b1");
    I.Drag("#pb2").To("#b2");
    I.Drag("#pboth1").To("#b3");
    I.Drag("#pboth2").To("#b4");
    I.Drag("#pt1").To("#pt2");
    I.Drag("#pboth1").To("#pb2");
}

Easily runs against IE (or if I.Use is excluded, the default browser for the provider). But what if you wanted to test it against Chrome?

You could do something like…

private void DragDropTestActions()
{
    I.Open("http://developer.yahoo.com/yui/examples/dragdrop/dd-groups.html");
    I.Drag("#pt1").To("#t2");
    I.Drag("#pt2").To("#t1");
    I.Drag("#pb1").To("#b1");
    I.Drag("#pb2").To("#b2");
    I.Drag("#pboth1").To("#b3");
    I.Drag("#pboth2").To("#b4");
    I.Drag("#pt1").To("#pt2");
    I.Drag("#pboth1").To("#pb2");
}

[TestMethod]
public void Drag_Drop_Test()
{
    I.Use(BrowserType.InternetExplorer);
    DragDropTestActions();
}

[TestMethod]
public void Drag_Drop_Test_Chrome()
{
    I.Use(BrowserType.Chrome);
    DragDropTestActions();
}

This isn’t necessarily wrong, and in some situations might even be required. But we can do better.

[TestMethod]
public void Drag_Drop_Test()
{
    I.Record();
    I.Open("http://developer.yahoo.com/yui/examples/dragdrop/dd-groups.html");
    I.Drag("#pt1").To("#t2");
    I.Drag("#pt2").To("#t1");
    I.Drag("#pb1").To("#b1");
    I.Drag("#pb2").To("#b2");
    I.Drag("#pboth1").To("#b3");
    I.Drag("#pboth2").To("#b4");
    I.Drag("#pt1").To("#pt2");
    I.Drag("#pboth1").To("#pb2");
    I.Execute(BrowserType.InternetExplorer, BrowserType.Chrome);
}

This last test will run the steps against IE first, and then Chrome. Any failures in either run will fail this test. Currently we do not continue to the next browser if an Exception is thrown or an Assertion/Expect fails. Looking for feedback on this behavior. Using these commands, it becomes incredibly easy to extend the scope of your automated testing to include other major browsers.

There is a trade-off though. The way this works, the actions are stored then executed when I.Execute is called. In general this isn’t an issue but it does mean you can’t count on passing data from one Command to another (something we strongly discourage in any case). A secondary issue is that, with only a single test, your test results are not as granular in scenarios where tools like Microsoft’s Test Manager product are being used.

Selenium WebDriver Provider

New to FluentAutomation, we’ve completed the Selenium WebDriver Provider.

Tests can now be written against Firefox and Chrome when Selenium is the chosen provider.

With some tweaks, all functionality is ‘standardized’ across the WatiN and Selenium providers. Some noticeable differences include WatiN running only against IE via Full Screen mode. Selenium can test against all the supported browsers and does not require Full Screen due to its native implementation of Drag-and-Drop.

To support Chrome via Selenium, you’ll need to download and include the ChromeDriver.exe in your project. Make sure it gets copied to the /bin folder when you execute tests.

FluentAutomation API

Announcing the new Fluent Automation API. A quick, easy to use library for automating web applications.

Here is an example of the new framework:

using FluentAutomation.API; using FluentAutomation.API.Enumerations; using Microsoft.VisualStudio.TestTools.UnitTesting;

namespace FluentAutomation.Samples
{
    [TestClass]
    public class LoginTests : TestClass
    {
        [TestMethod]
        public void Login_HappyPath()
        {
            I.Use(BrowserType.InternetExplorer);
            I.Open("http://www.yoursite.com/Login");

            // Validate expected strings on page
            I.Expect.This("Username").In("label.username-field");
            I.Expect.This("Password").In("label.password-field");

            // Perform login
            I.Enter("username").In("input.username-field");
            I.Enter("password").In("input.password-field");
            I.Click("input[type='button'].login");

            // Validate login successful
            // Forwarded correctly
            I.Expect.Url("http://www.yoursite.com/");
        }

        [TestMethod]
        public void Login_BadPassword()
        {
            I.Use(BrowserType.InternetExplorer);
            I.Open("http://www.yoursite.com/Login");

            // Validate expected strings on page
            I.Expect.This("Username").In("label.username-field");
            I.Expect.This("Password").In("label.password-field");

            // Perform login
            I.Enter("username").In("input.username-field");
            I.Enter("wrong password").In("input.password-field");
            I.Click("input[type='button'].login");

            // Validate login successful
            // Error shown
            I.Expect.This("Username/password not correct.").In("div.error");
        }
    }
}

Much more information coming soon!

NOTE: This is very old. Several changes have been made to the API since this was posted. Please reference the Docs when writing new tests.
